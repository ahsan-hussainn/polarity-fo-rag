# ADR-0003: Dataset schema mirrors the FO-MAX sample, with a verifiability-first contact layer

- **Date:** 2026-07-11
- **Status:** Accepted

## Context

Observed (fact): The provided reference file `FO-MAX-data-sample-2.0.xlsx` has 32 columns and ~112 rows.
Every high-value contact cell is redacted as `Hidden` (primary/secondary email, email validation code,
email code explanation, email quality assessment, phone, LinkedIn). Everything around them (firm name,
thesis, sectors, description, address, contact name + title) is shown. The sheet also carries a per-record
`Data Completion Score` and a `URL Quality` rating.

Inference (not certain): the redaction marks where the paid value sits. The contact layer, and specifically
the email + its validation, is the product. Their column design already encodes a validation layer:
`email -> validation code -> code explanation -> quality assessment`, duplicated for a secondary email.

Observed (fact): the task scores each record on (1) actionability (could a fund manager act today) and
(2) verification (every high-value cell carries its basis; sample-checked; a cell that fails its stated
verification costs more than an honest blank; a guessed value dressed as verified is disqualifying).

## Decision

Adopt a schema that mirrors FO-MAX's structure so our output is directly comparable to their reference, and
extend the contact layer so verifiability is a first-class, machine-populated part of every high-value cell:
each such cell gets a paired `*_source` (where it came from) and `*_verification` (method + code + confidence)
field. Carry a per-record completeness/quality score like theirs.

## Options considered

- **Mirror FO-MAX + explicit per-cell source/verification (chosen).**
- **Invent our own schema from scratch:** rejected. They gave a reference; matching it makes evaluation
  direct and signals we read the sample. Original *records* are required, not an original *schema*.
- **Flat schema without per-cell provenance:** rejected. "Verified with no method is a claim, not a fact."
  Provenance has to live in the data, not only in a writeup.

## Why this over the others

Matching their shape removes a translation cost for the evaluator and lets us beat the sample where it is
weak (e.g. their Walton record's contact is an "Accounting Manager," not a principal). Pairing every
high-value cell with a source and a verification method is the concrete mechanism for the verification score,
and it is what turns "a spreadsheet" into "decision-grade."

## Assumptions and risks

- Assumption: the email-validation apparatus in their schema is the intended core value. Strong inference
  from the redaction pattern, not confirmed. If wrong, we have still built the most defensible thing.
- Risk: full per-cell provenance is heavy to populate automatically for 50 records in the window. Mitigation:
  provenance is generated by the pipeline as a byproduct of discovery/enrichment, not added by hand.
- Ethics/legal: sourcing principal emails/phones is PII. We restrict to legitimately public sources and mark
  honest blanks rather than guess. Tracked as its own decision (to be recorded when the sourcing pipeline is
  designed).

## What would change this

If populating the full contact + provenance layer proves infeasible at quality within the window, we narrow
the verified contact scope (e.g. firm-level verified contact only) and say so explicitly, rather than
shipping guessed cells.
